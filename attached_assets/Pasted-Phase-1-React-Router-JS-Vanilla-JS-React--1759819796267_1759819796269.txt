Phase 1: 기반 구축

요약: React Router + 기본 레이아웃 도입, 기존 JS 공존

✅ 좋은 점

“Vanilla JS와 React 공존” 접근은 맞아.
한 번에 갈아엎지 않고, React로 진입점만 옮기는 게 가장 안전함.

라우팅 구조를 App.jsx에서 잡는 건 필수.

⚠️ 보완해야 할 점

iframe 방식 공존은 TL처럼 SPA 기반 구조에선 비효율적임.
대신 “React 안에서 legacy 모듈을 직접 mount”하는 구조가 나음.
예:

useEffect(() => {
  import('/Frontend/TLG/pages/main/renderMap.js').then(mod => mod.render());
}, []);


→ 이러면 React가 “전역 main 컨테이너”만 제공하고 JS는 안에서 그대로 작동.

Phase 2: 인증/로그인 전환

요약: 로그인/회원가입을 React로 교체

✅ 좋은 순서

로그인은 독립적이라, 공존 구조 안에서도 React로 완전히 분리 가능.

로그인 페이지는 UI도 단순하고 API만 연결하면 되니까 첫 대상으론 최적.

💡 추가 제안

지금 authService.js는 이미 Express 라우트(/api/auth/login)랑 맞물려 있잖아?
→ 그걸 React 쪽 features/auth/services/로 import해서 재활용 가능.

이 단계에서 React Hook Form + Zod 조합 도입 추천.

Phase 3: 메인/지도 페이지 전환

요약: 가장 복잡한 화면 (지도, 매장 패널, 검색)

✅ 좋은 방향

상태 관리, API 캐싱, UI 상태가 복잡하므로 React Query 도입 타이밍이 딱 맞음.

⚠️ 주의점

TLA-MAP (카카오 지도) 모듈은 React로 바로 옮기지 말고,
기존 JS SDK 로더를 재활용하는 방식으로 감싸야 해.

useEffect(() => {
  const script = document.createElement("script");
  script.src = "https://dapi.kakao.com/v2/maps/sdk.js?...";
  document.body.append(script);
}, []);


React에서 지도는 비동기 로드 후 <div id="map" />에 attach하는 구조로 전환해야 함.

Phase 4: 매장 상세 전환

요약: 탭 구조 (홈, 메뉴, 리뷰, 정보, 단골)

✅ 구조는 완벽히 맞음

components/와 hooks/ 분리 방식 아주 좋아.

TL의 renderStore 구조는 React 컴포넌트화하기 가장 이상적인 케이스야.

💡 추천

React Router의 중첩 라우트(Outlet)로 탭 전환 구현하면 깔끔함.

리뷰는 TLR(TL Retention) 구조니까 ReviewTab은 features/review/로 분리 관리해도 좋아.

Phase 5: 주문/결제 전환

요약: TLL(주문) + Toss(결제) 리액트화

✅ 핵심 로직 맞음

Zustand로 전역 주문 상태 관리 → 맞음.

React Query로 결제 API 요청 → 맞음.

모듈 단위도 현실적.

⚠️ 주의점

Toss SDK는 React Hook 기반으로 감싸줘야 함.

useEffect(() => {
  const script = document.createElement("script");
  script.src = "https://js.tosspayments.com/v1/payment-widget";
  document.body.append(script);
}, []);


결제 완료 후 React Router로 navigate('/pay/success') 처리로 통합.

Phase 6: 마이페이지 전환

✅ 단순 구조라서 React 전환 쉬움.
리뷰, 포인트, 쿠폰 등은 React Query로 캐싱하면 완벽.

💡 추가 팁

TLR(리텐션) 기능을 여기서 연동할 수 있음.
즉, React에서 리뷰 작성, 쿠폰 확인, 포인트 내역까지 한 화면에 구성 가능.

Phase 7: POS/KDS/Admin

✅ 별도 React 프로젝트로 분리하는 게 맞음.

POS/KDS는 B2B 영역이라 라우트만 분리하는 것보다 독립 빌드가 훨씬 나음.

TL 구조상 /pos, /kds, /admin은 완전 별 UI 흐름이라 독립 배포에 유리.

💡 추가 제안

이후 Electron으로 POS 빌드하는 경우 React 구조를 그대로 재활용 가능.

/shared 컴포넌트는 npm local 패키지로 분리 고려 (공통 UI/훅/유틸).

Phase 8: 최적화 및 완성

✅ 최종 정리단계로 맞음.

Code Splitting, Lazy Loading, Test 도입은 타이밍 정확.

단, Next.js 전환은 TL 같은 앱에는 굳이 필요 없음.
TL은 SEO보다 앱 UX가 중요하니까 SSR은 오히려 오버스펙.

✅ 결론 (요약 정리)

리플릿 AI가 제안한 React 전환 로드맵은 전체 방향은 매우 옳고,
TL 구조 기준으로도 논리적으로 실행 가능함.

다만, **Phase 1(공존 방식)**만 “iframe” 대신
“React 컨테이너 안에 기존 JS 모듈 마운트” 방식으로 수정하는 게 현실적으로 낫고,
Next.js SSR은 TL에 필요 없음.

나머지는 그대로 따라가도 무방함 — 3~4개월 로드맵으로 충분히 가능.
